# PBPK MCP v0.4.4

## Release Summary

- `PBPK MCP v0.4.4: release-gate convergence, live PKML export hardening, and publication-control tightening`

## Overview

PBPK MCP `v0.4.4` is a patch release focused on publication readiness rather than a new scientific contract.

This release makes the public-facing path more defensible by:

- converging local and workflow release evidence on one retained preflight summary
- reducing the slowest live `.pkml` release-gate hotspot without widening claims
- tightening audit, signoff-history, and GitHub publication controls

## Added

- retained `public_release_preflight_summary.json` and matching workflow evidence on the `Model Smoke` path
- a dedicated review-signoff index backfill utility for legacy audit history
- documented local S3-compatible audit smoke coverage as part of release-prep and operator proof

## Changed

- the live `.pkml` release gate now proves sidecar-backed OECD report export without forcing a full runtime parameter-table preview
- the source-overlay runtime now does less eager startup work in app/auth/audit/tool-registry layers, which reduces redeploy friction during release-prep
- release docs now treat the live GitHub `main` ruleset, retained preflight summary, and workflow artifacts as first-class publication evidence
- the `Model Smoke` workflow now runs the consolidated preflight entrypoint and uploads the same retained summary used in local release-prep

## Fixed

- aligned the live smoke timeout budget with the real `workspace_model_smoke.py --include-population` runtime so the pre-public gate no longer fails on a harness false negative
- restored live contract-artifact convergence after regeneration by redeploying against the refreshed bundled manifest set
- fixed the last async release-gate false negative by treating Celery `RECEIVED` work as running and extending the reference-model async poll budget to match the measured packaged runtime
- fixed source-overlay contract drift so regenerated `docs/architecture/*manifest.json` artifacts stay aligned with the live overlay runtime during release-prep

## What This Release Does Not Claim

`v0.4.4` does **not** widen the scientific or regulatory scope of PBPK MCP.

It does not:

- make research-use examples regulatory-grade
- add native OSPSuite population execution
- move reverse dosimetry, IVIVE, or cross-domain uncertainty synthesis into PBPK MCP

The release strengthens proof, control, and trust-surface clarity around the existing `pbpk-mcp.v1` boundary.

## Verification

The release state was checked with:

```bash
./.venv/bin/python scripts/check_release_metadata.py
./.venv/bin/python scripts/check_distribution_artifacts.py
./.venv/bin/python scripts/generate_contract_artifacts.py --check
./scripts/deploy_source_overlay_stack.sh
./.venv/bin/python scripts/release_readiness_check.py --base-url http://127.0.0.1:8000 --skip-unit-tests --auth-dev-secret pbpk-local-dev-secret-32bytes-long
./.venv/bin/python scripts/public_release_preflight.py --auth-dev-secret pbpk-local-dev-secret-32bytes-long
```

Those checks confirmed:

- release metadata alignment at `0.4.4`
- packaged `sdist`/wheel integrity
- current contract-manifest and release-bundle integrity
- a live runtime whose contract-manifest artifact inventory matches the regenerated published bundle
- a passing retained end-to-end public preflight summary

## Why It Matters

PBPK MCP `v0.4.4` is a better public release because it is easier to verify and harder to mispublish.

The repo now has a clearer answer to three questions that matter before a public tag:

- what exact checks define “ready to publish”
- what artifact proves those checks passed
- what GitHub controls must be active before code reaches `main`
