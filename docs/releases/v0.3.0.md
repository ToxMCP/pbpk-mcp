# PBPK MCP v0.3.0

Suggested GitHub release title:

- `PBPK MCP v0.3.0: patch-first contract convergence`

Suggested short release summary:

- PBPK MCP `v0.3.0` makes the documented dual-backend, discovery-first, OECD-oriented workflow the tested live contract of the current patch-first runtime.

## Highlights

- converges the live tool surface around the documented workflow from discovery through results and OECD report export
- adds a shared runtime patch manifest so the Docker image build and the hot-patch flow apply the same file set
- tightens conversion-only format handling for `.pksim5` and `.mmd`
- strengthens release-readiness checks with tool-catalog, discovery/resource parity, and explicit conversion-boundary assertions

## Included In This Release

### Contract convergence

- keeps `patches/` as the canonical implementation layer for this stage
- preserves `pbpk-mcp.v1` as the normalized response contract
- verifies the documented workflow on the live stack through `/mcp/list_tools`, `/mcp/resources/models`, deterministic runs, and OECD report export

### Patch-first deployment hardening

- introduces `scripts/runtime_patch_manifest.py` as the shared runtime patch map
- introduces `scripts/install_runtime_patches.py` so image builds and hot patches apply the same contract surface
- updates `scripts/apply_rxode2_patch.py` to patch both API and worker by default
- updates `scripts/deploy_rxode2_stack.sh` to recreate the stack and immediately reapply the current workspace patch set

### Runtime policy tightening

- direct `.pksim5` loading is rejected with explicit guidance to export to `.pkml`
- direct Berkeley Madonna `.mmd` loading is rejected with explicit guidance to convert to `.pkml` or an MCP-ready `.R` module

## Limitations To Call Out Publicly

- the current convergence stage is still patch-first for the local runtime; a pure packaged `src/` migration is deferred
- runnable and qualified remain separate concepts
- Berkeley Madonna `.mmd` remains a conversion source, not a runtime backend
- `.pksim5` remains a project/export source, not a direct runtime format

- `discover_models`

- `load_simulation`
- `validate_simulation_request`
- `run_simulation` / `run_population_simulation`

- `export_oecd_report`

Important boundaries remain explicit:

- `.pkml` and MCP-ready `.R` are the supported runtime formats
- `.pksim5` must be exported to `.pkml` before runtime use
- Berkeley Madonna `.mmd` is a conversion source, not a runtime backend
- executable does not mean scientifically qualified

This release does not introduce new BER or broader NGRA domain objects yet. It focuses on making the published PBPK MCP contract real, reproducible, and release-ready.
