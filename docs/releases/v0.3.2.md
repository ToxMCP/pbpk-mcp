# PBPK MCP v0.3.2

## Release Summary

- `PBPK MCP v0.3.2: local deployment startup hotfix`

## Overview

PBPK MCP `v0.3.2` is a deployment-focused patch release over `v0.3.1`.

It fixes a local runtime startup issue in the patch-first Docker deployment:

- the compose stack no longer sets `PYTHONPATH=/app/src`
- the mounted `src/` tree therefore no longer shadows the installed `mcp_bridge` package
- the API and Celery worker now start cleanly against the patched installed package, which is the intended runtime model for the current patch-first stage

## Fixed

- restored startup for:
  - `mcp_bridge.app:create_app`
  - `mcp_bridge.services.celery_app.celery_app`
- removed a deployment-path mismatch where the mounted `src/` directory could override the installed runtime package even though `patches/` plus the patch installer remain the canonical live implementation path

## Why It Matters

This release does not add new MCP features.

It matters because a qualification-aware platform is only credible if the documented local deployment model actually boots and stays aligned with the tested runtime contract.

`v0.3.2` makes the local Docker deployment consistent with the current patch-first architecture:

- patched runtime files are copied into the installed package path
- the live API imports from that patched installed package
- the mounted `src/` tree is no longer given import precedence

## Verification

The hotfix should be validated with:

```bash
docker compose -f docker-compose.celery.yml config
make runtime-contract-test
python3 scripts/release_readiness_check.py --skip-unit-tests
```

For the public release, GitHub Actions CI should also pass on `main`.
