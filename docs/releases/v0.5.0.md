# PBPK MCP v0.5.0

## Release Summary

- `PBPK MCP v0.5.0: auth replay enforcement, parameter-governance parity, and release metadata convergence`

## Overview

PBPK MCP `v0.5.0` is a control-surface and release-hygiene update focused on making the shipped runtime behave the same way its public contract now claims.

This release makes the operational boundary more defensible by:

- rejecting replay of already-seen bearer tokens that carry `jti` claims
- upgrading `/mcp` to MCP `2025-11-25` JSON-RPC behavior while preserving REST helper endpoints
- exposing published artifacts and live catalogs as standard MCP resources with `pbpk://` URIs
- aligning REST and MCP parameter-write governance on one snapshot/audit path
- moving canonical PBPK tool modules to `mcp_bridge.pbpk_tools` with a temporary deprecated `mcp.*` shim
- making `relative` parameter updates behave as declared by the published contract
- completing the public `v0.5.0` metadata alignment across package, compose, README, and release-note surfaces

## Changed

- JWT validation now treats reuse of an already-seen `jti` token identifier as an authentication failure instead of a harmless repeat validation
- JSON-RPC success/error responses now use clean MCP-compliant shapes, `id: null` is rejected, accepted notifications and client responses return empty HTTP `202`, and `GET /mcp` returns `405` until SSE is implemented
- `initialize` now defaults to `2025-11-25`, negotiates legacy `2025-03-26`, and advertises standard tools/resources capabilities
- `tools/list` now includes `inputSchema`, `outputSchema`, and standard annotations; `tools/call` now returns `structuredContent`
- `resources/list`, `resources/read`, and `resources/templates/list` now expose contract and runtime resources through `pbpk://` URIs
- MCP streamable HTTP now supports `MCP_ALLOWED_ORIGINS` and optional `MCP_STRICT_TRANSPORT=true`
- the supported Python floor is now `3.11+`
- REST `/set_parameter_value` now returns the same governance payload and triggers the same snapshot/audit side effects as MCP `call_tool`
- `set_parameter_value(updateMode="relative")` now resolves an effective target value before guardrail validation, audit recording, and persistence
- the default local compose profile now uses canonical `ADAPTER_MODEL_PATHS` configuration, while legacy alias warnings remain in place until `v0.6.0`

## Fixed

- removed the transport split where REST parameter updates skipped snapshot creation, sweep/audit hooks, and governance response data
- fixed the silent contract bug where `relative` parameter updates overwrote the current value instead of applying a delta
- restored release-story convergence so the runtime and documentation no longer advertise `0.4.4` after the `v0.5.0` update

## What This Release Does Not Claim

`v0.5.0` does **not** widen the scientific or regulatory scope of PBPK MCP.

It does not:

- make research-use examples regulatory-grade
- add native OSPSuite population execution
- move reverse dosimetry, IVIVE, or cross-domain uncertainty synthesis into PBPK MCP

The release tightens runtime control, transport parity, and publication accuracy around the existing `pbpk-mcp.v1` boundary.

## Verification

The release state was checked with:

```bash
./.venv/bin/pytest -q tests/test_security_posture.py tests/test_parameter_governance.py tests/test_config_contract.py tests/test_deployment_profiles.py
./.venv/bin/pytest -q tests/test_cross_parameter_consistency.py tests/test_run_parameter_consistency_check.py tests/test_parameter_bounds.py
./.venv/bin/python scripts/check_release_metadata.py
./.venv/bin/python scripts/generate_contract_artifacts.py --check
```

Those checks confirm:

- replay rejection for reused `jti` token identifiers
- REST/MCP parity for parameter-write governance surfaces
- working `relative` parameter updates
- release metadata alignment at `0.5.0`

## Why It Matters

PBPK MCP `v0.5.0` is a cleaner release because the public contract, runtime behavior, and release story now agree on the same boundary.
